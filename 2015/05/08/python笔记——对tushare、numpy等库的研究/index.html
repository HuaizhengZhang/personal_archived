python笔记——对tushare、numpy等库的研究

python确实是门很方便的语言，尤其是对于金融数据的统计计算，做的人很多，使网上可利用的库相当多。

按老师的要求，要拿一些股票数据来用，就正好学着用一下tushare、numpy、pandas这几个库。

上来就是一个大坑，tushare用的pandas的DataFrame格式的数据表示，可以方便的使用pandas的一切语法，to_csv等等，对于不太习惯这个格式的我，还是选择了在对文件中的数据用numpy的方式提取，在内存中的数据处理使用python自己的list，感觉笨笨的，但是自己用的比较顺手。

然后就是第二个坑，对于tushare所提供的get_h_data这个函数，开始日期和结束日期强制使用的是字符串格式，而为了便于对时间进行准确的计算，最好还是使用python自带的datetime这个库，日期时间计算，加加减减，突出一个方便。所以就不得不来来回回的在strftime，strptime之间进行转换，一不留神就会出现错误。并且这个函数对于起止日期超过一年的获取很容易出错，不得不分段做起，很麻烦，也不geek。在对时间分段的时候切记要把现在的时间作为变量进行处理，倒退时间，不然在csv文件的时候会出现一个有趣的逻辑错误。

做到最后，还是后悔用tushare了，整体程序不健壮，不过人家作者也辛苦拿出来给大家开源使用，单纯用其中几行代码获取数据然后进行分析，还是比较方便，但真正的想快速的拿到全部股票数据，还是需要自己慢慢爬，不过这也促成了自己在写代码以来第一次正了八经的用try这条语句，挺好用。

从600606开始是什么鬼？tushare本身莫名其妙的错有点多

果然还是代码的思维比较重要，虽然最近几个周没咋写，但是看了些书，整理了下思路，确定自己在代码过程中要达成的目标，写起来比以前顺手多了。而python这个语言快速开发这个特性也确实比较帮助我这个比较喜欢敏捷开发的能够快速建立模型，把思路理清楚，然后迭代开发。

下面应该是要重写了，tushare对于获取日K线数据比较方便，但整体数据要保持和原来学长需要调用的数据的结构相同，写起来就是突出一个麻烦。

接下来应该是用曾经用过的Scrapy框架爬取数据，用numpy进行处理，然后写入csv文件，最后在把单线程改为多线程处理，提高数据下载速度，不知道能不能比C++写的快，这两天有些事情耽搁不能写，暂且记录下自己的思路，清空大脑，下个周好好写一下。
